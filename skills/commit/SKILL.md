# Plan and Execute Atomic Commits

Analyse all unstaged/staged changes and split them into the right number of atomic commits. Each commit message must convey the actual work done — not the file changed or test type, but the feature or fix it delivers.

## Step 0 — Get current git state
Run via Bash tool: `git branch --show-current && git status --short && echo "---" && git diff --stat HEAD | tail -5`

## Usage

```
/commit          # analyse all uncommitted changes and plan commits
/commit staged   # only commit what is already staged (skip planning)
```

## Commit Message Rules

- **One line, max ~72 characters**
- **Start with a lowercase verb**: `add`, `fix`, `implement`, `wire`, `extract`, `replace`, `remove`, `expose`, `validate`, `support`
- **Describe the feature/behaviour, not the file or test**: say what the system now does, not what file was touched
- **No ticket ID** — it's in the branch name already
- **No "Co-authored-by"**, "Generated by", or any Claude attribution

**Good — describes the work:**
```
implement rate limiting on exp_execute_query using token bucket
add OpenSearch pagination support to entity search tool
expose entity archive endpoint through exp_manage_entity_lifecycle
fix auth token missing from downstream context service calls
validate project_id as string to prevent float precision loss
support headless execution mode for entity lifecycle tool
```

**Bad — describes the file or test type:**
```
add unit tests
update service.py
fix bug
add new tool
```

## Steps

### 1. Get the full picture of uncommitted work

```
git diff --staged --stat
git diff --stat
git status --short
git branch --show-current
```

If nothing at all is changed (staged or unstaged), stop and say so.

### 2. Read the full diff

```
git diff HEAD
```

If the diff is very large (>500 lines), read `git diff HEAD --stat` first and then read individual files selectively.

### 3. Group changes into logical commits

Cluster changed files into groups where each group represents one coherent unit of work.

**Grouping rules:**
- A new tool file + its service + its DTOs = one commit
- A new tool file + its tests = one commit (feature and its proof)
- Config/env changes that enable a new tool = separate commit if standalone, otherwise bundle
- A bug fix independent of other changes = its own commit
- Refactors that don't change behaviour = their own commit
- Dependency additions that serve a specific feature = bundle with that feature

**Do not split arbitrarily.** If 10 files all implement one feature, that's one commit.

### 4. Present the commit plan

Print the plan **before executing anything**:

```
## Commit Plan — <branch name>

N commits proposed:

1. implement exp_query_optimizer tool with OpenSearch template routing
   Files: src/tools/query_optimizer.py, src/services/query_optimizer_service.py, ...

2. add EXP_QUERY_OPTIMIZER_URL to config and env example
   Files: src/env_config.py, .env.example
```

Wait for confirmation. Accept adjustments: "merge 1 and 2", "split 3", "skip 2", "swap order of 1 and 3". Or "yes" to execute.

### 5. Execute confirmed commits sequentially

For each commit:
1. `git add <file1> <file2> ...`
2. `git commit -m "<message>"`
3. Show: `git log --oneline -1`

After all commits, print a summary of all commits made.

### 6. Staged-only mode (`/commit staged`)

Skip steps 1–4. Read `git diff --staged` only, generate exactly one commit message, confirm, and commit.
